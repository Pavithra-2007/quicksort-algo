Recursive implementation
// Find k-th smallest in arr[lo..hi] function quickSelectRec(arr, lo, hi, k): if lo == hi: return arr[lo] // base case — single element p = partition(arr, lo, hi) if k == p : return arr[k] // pivot IS the answer elif k < p : return quickSelectRec(arr, lo, p-1, k) else : return quickSelectRec(arr, p+1, hi, k) medianRec(arr): k = ⌊arr.length / 2⌋ return quickSelectRec(arr, 0, arr.length-1, k)
Iterative (non-recursive) implementation
// Same result — while loop replaces the call stack function quickSelectIter(arr, k): lo = 0 hi = arr.length - 1 while lo < hi: p = partition(arr, lo, hi) if p == k : break elif p < k : lo = p + 1 else : hi = p - 1 return arr[k] // Lomuto partition — pivot = last element function partition(arr, lo, hi): pivot = arr[hi]; i = lo - 1 for j = lo to hi-1: if arr[j] <= pivot: i++ swap(arr, i, j) swap(arr, i+1, hi) return i + 1
Comparison
Recursive
Calls itself with a smaller subproblem. Elegant and readable. Uses O(log n) call-stack depth on average; O(n) worst case — potential stack overflow on large sorted inputs.
Iterative
Updates lo/hi pointers in a loop. Preferred in production — O(1) auxiliary space, no stack overflow risk on adversarial input. Both versions do identical comparisons.

Recurrence intuition
Each call partitions O(n) elements and recurses into only one half:

T(n) = T(n/2) + O(n) → O(n) (Master theorem, case 3)
Compare with merge sort: T(n) = 2·T(n/2) + O(n) → O(n log n)
QuickSelect recurses into one branch, not both — that's the entire advantage.
